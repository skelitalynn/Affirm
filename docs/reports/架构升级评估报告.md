## 二、升级风险评估

### 1. LangGraph Agent 重构

**是否适合当前项目？**

当前项目的对话逻辑极为简单：用户消息 → 拼接上下文 → 一次 LLM 调用 → 返回。LangGraph 的优势在于多步推理、条件分支、工具调用。当前项目不存在这些需求。

| 分析维度 | 结论 |
|---------|------|
| 当前对话复杂度 | 单步，无条件分支 |
| 工具调用需求 | 目前无（若接 RAG 则有价值） |
| 状态机需求 | 仅需简单上下文追踪 |
| LangGraph 适用场景 | 多步 Agent、ReAct 推理、复杂工作流 |

**需要修改的模块：**
- `AIService.generateResponse()` — 需重写为 LangGraph Graph 节点
- `TelegramService._processSingleMessage()` — 调用方式需改变
- 需新增 `src/agents/` 目录，定义 State、Nodes、Edges

**预计改动规模：** 新增约 200-400 行代码，`AIService` 需完全重写（保留接口签名可降低下游改动）

**主要风险：**
- LangGraph JS 版本（`@langchain/langgraph`）相比 Python 版本 API 尚不稳定
- 当前业务逻辑太简单，重构收益低、引入依赖成本高
- 如 RAG 同步接入，则 LangGraph 有价值（RAG Node + LLM Node 的两步流）

### 2. Embedding Provider 独立

**核心问题定位（代码层面）：**

```javascript
// services/embedding.js:14-17 — 问题所在
this.openai = new OpenAI({
    apiKey: config.ai.apiKey,    // 用的是主 AI Provider 的 Key
    baseURL: config.ai.baseURL   // 用的是主 AI Provider 的 URL
});
// config.js:44 → baseURL = 'https://api.deepseek.com/v1'
// DeepSeek 无 /embeddings 端点 → API 调用必然失败
```

**接口设计方案：**

在 `config.js` 中新增独立 embedding 配置块：

```javascript
embedding: {
    provider:   process.env.EMBEDDING_PROVIDER || 'openai',
    apiKey:     process.env.EMBEDDING_API_KEY,
    baseURL:    process.env.EMBEDDING_BASE_URL || 'https://api.openai.com/v1',
    model:      process.env.EMBEDDING_MODEL || 'text-embedding-3-small',
    dimensions: parseInt(process.env.EMBEDDING_DIMENSIONS) || 1536
}
```

`EmbeddingService` 改为读取 `config.embedding`，与 `config.ai` 完全解耦。

**是否影响数据库 Schema？**

| 情况 | 影响 |
|------|------|
| 仍使用 768 维模型（如 nomic-embed-text） | 无影响 |
| 改用 OpenAI `text-embedding-3-small`（1536 维） | 需迁移 messages 和 knowledge_chunks 的 vector 列 |
| 改用 `text-embedding-3-large`（3072 维） | 需迁移 + 重新生成所有历史向量 |

**历史数据迁移风险：** 如果存在历史向量数据（维度 768），切换模型后必须重新生成全部 embedding，否则相似度计算无意义。

### 3. Webhook 架构

**与 Polling 的本质差异：**

| 特性 | Polling (`{ polling: true }`) | Webhook |
|------|-------------------------------|---------|
| 连接方向 | Bot 主动轮询 Telegram | Telegram 主动推送 |
| 延迟 | ~1-3 秒 | < 100ms |
| 进程要求 | 单进程即可 | 需要公网 HTTPS 端点 |
| 水平扩展 | 不可（多进程重复消费） | 可（配合队列） |
| 本地开发 | 直接运行 | 需 ngrok / 内网穿透 |
| 冲突风险 | 可能与 webhook 冲突 | 无 |

**实现所需改动：**

1. `telegram.js` 中 `new TelegramBot(token, { polling: true })` → 改为 webhook 模式或接入 Express 路由
2. 需要 Express 接收 `POST /webhook/:token` 请求
3. 必须有 HTTPS：Telegram 仅向 HTTPS 端点发送 webhook（自签证书需配置）
4. 调用 `bot.setWebHook(webhookUrl)` 注册 URL

**是否需要反向代理？**

生产环境建议：`Nginx → Node.js`（Nginx 处理 SSL 终止）
开发环境：`ngrok` 临时隧道足够

### 4. Redis Queue

**当前 MessageQueue 的缺陷：**

```javascript
// utils/message-queue.js:10
this.userQueues = new Map();  // 纯内存，进程重启消失
// 无法跨多个 Node.js 进程共享
// Webhook 部署时若多实例则队列失效
```

**Redis Queue 替代方案（推荐 BullMQ）：**

```
MessageQueue.enqueue(userId, fn)
    ↓ 替换为
BullMQ Queue.add('processMessage', { userId, chatId, text })
    ↓
BullMQ Worker 消费队列
```

**是否需要任务持久化？**

| 场景 | 是否需要持久化 |
|------|--------------|
| 消息处理失败需重试 | 是（BullMQ 原生支持） |
| 进程重启后继续处理 | 是 |
| 审计/追踪任务历史 | 是 |
| 仅需并发控制 | 否（但内存队列仍有重启丢失问题） |

建议开启持久化（BullMQ 默认使用 Redis 持久化）。

**依赖新增：** `bullmq` + `ioredis`，需要运行 Redis 实例（可通过 Docker Compose 添加）

### 5. RAG Integration

**现状：**
`Knowledge.semanticSearch()` 和 `Message.semanticSearchByText()` 已实现，但 `_processSingleMessage()` 中完全未调用。

**接入点（`telegram.js` 第 3-4 步上下文构建处）：**

```javascript
// 当前（仅时序消息）：
const recentMessages = await Message.getRecentMessages(user.id, contextLimit);
const context = {
    user: { ... },
    userMessage,
    recentMessages
};
```

**RAG 接入后的构建逻辑：**

```javascript
// Step 3a: 语义检索相关知识片段
const relevantKnowledge = await Knowledge.semanticSearch(
    userMessage, userId, 5, 0.6
);

// Step 3b: 语义检索相关历史消息
const semanticMessages = await Message.semanticSearchByText(
    userMessage, userId, 3, 0.65
);

// Step 4: 扩展上下文
const context = {
    user: { ... },
    userMessage,
    recentMessages,       // 最近 N 条时序消息
    relevantKnowledge,    // RAG: 相关知识片段
    semanticMessages      // RAG: 相关历史消息
};
```

**Prompt 构造（`AIService.prepareMessages` 中扩展）：**

```javascript
let ragContext = '';
if (context.relevantKnowledge?.length > 0) {
    ragContext = '\n\n相关知识背景：\n' +
        context.relevantKnowledge
            .map(k => `- ${k.content}`)
            .join('\n');
}

const systemPrompt = `你是一个有帮助的显化导师...
${ragContext}
请保持温暖、鼓励的语气...`;
```

**前提：** Embedding Provider 必须先独立，否则 RAG 无法正常工作（DeepSeek 无 embedding）

### 架构决策说明

**BullMQ concurrency=1（全局串行）**

当前实现使用单 Worker 实例、`concurrency: 1`，所有用户消息全局串行处理。这与原内存队列"按用户串行"不同，是有意的折衷选择：
- 优点：实现简单，避免 BullMQ Pro 依赖（per-group concurrency 是付费功能）
- 影响：高并发时不同用户的消息也会互相等待（但对于个人/小团队 Bot 影响可忽略）
- 改进路径：参见下方"新发现问题"

**内存降级策略**

若 Redis 不可用（连接失败），系统自动降级为原内存队列实现，行为与 Phase 1 完全一致。降级时控制台输出警告，不影响 Bot 运行。

**`attempts: 1`（不重试）**

BullMQ 任务不重试。原因：`_processSingleMessage` 内部已处理错误并向用户发送提示消息。若启用重试（attempts > 1），用户会收到重复的错误消息。改进路径：区分"用户侧错误"和"基础设施错误"，仅对后者重试。

### 新发现问题（待后续处理）

以下问题在 Phase 2 实施过程中发现，暂不修改，记录供后续评估：

**P2-1: BullMQ per-user 串行化缺失**

- **现象**：BullMQ `concurrency: 1` 将所有消息全局串行，而不是"同一用户内串行、不同用户并发"
- **影响**：用户 A 的消息会阻塞用户 B 的消息处理
- **解决方案**：
  1. 升级 BullMQ Pro（支持 Job Groups，官方 per-group 并发）
  2. 或：提高 `concurrency`，在 Worker 内用 Redis SETNX 实现 per-user 互斥锁
- **优先级**：低（Bot 规模小时不影响体验）

**P2-2: 多实例部署时 bot.sendMessage 耦合**

- **现象**：BullMQ Worker 处理任务时通过 `this.bot.sendMessage()` 发送回复，`this.bot` 是当前进程的 TelegramBot 实例
- **影响**：若 Worker 和 Bot 分离为独立进程/容器，Worker 无法访问 `this.bot`
- **解决方案**：抽离 `sendMessage` 调用，Worker 完成处理后通过 Telegram Bot API 直接 HTTP 调用，或使用共享的 Bot 客户端单例
- **优先级**：中（水平扩展时必须解决）

**P2-3: BullMQ 重试与用户通知冲突**

- **现象**：当前 `attempts: 1`（不重试），无法自动恢复基础设施故障（DB 短暂中断等）
- **影响**：基础设施瞬时故障会导致消息处理失败，用户需重新发送
- **解决方案**：区分异常类型（`DatabaseError`、`AIError` 等），仅对基础设施错误启用重试，用户通知仅在最终失败时发送
- **优先级**：中

**P2-4: Webhook 注册幂等性**

- **现象**：每次 `start()` 都调用 `bot.setWebHook(url)`，若 URL 未变则为幂等操作；但若 URL 变更（如更新域名后未删除旧 webhook），Telegram 可能有短暂的双注册状态
- **解决方案**：启动时先 `getWebHookInfo()`，仅在 URL 不一致时才重新注册
- **优先级**：低
